# Vault — Encryption & Decryption

The vault uses a **double encryption layer (E2EE + server-side)**. Data never reaches the server in plaintext — the client always sends everything already encrypted (L1), and the server only adds a second layer (L2) before persisting. Without the user's master key, which never travels to the server, the original content is inaccessible.

---

## Layer Overview

```
[Original data]
      │
      ▼  (Layer 1 — client)
[AES-256-GCM with user's master key]
      │  → encryptedHeader  (base64)
      │  → encryptedBody    (base64)
      │  → clientIv         (base64)  ← IV generated by the client
      │
      ▼  (Layer 2 — server)
[AES-256-GCM with server master key]
      │  → encryptedHeader  (bytea in DB)  ← now doubly encrypted
      │  → encryptedBody    (bytea in DB)
      │  → serverHeaderIv   (bytea)
      │  → serverHeaderTag  (bytea)
      │  → serverBodyIv     (bytea)
      │  → serverBodyTag    (bytea)
      │  → clientIv         (bytea)  ← preserved for the client to decrypt later
```

---

## Server Algorithm

- **Algorithm**: `AES-256-GCM`
- **IV length**: 12 bytes (generated via `randomBytes(12)` per operation)
- **Key**: `cryptoConfig.masterKey` (environment variable `CRYPTO_MASTER_KEY`)
- **Implementation**: `src/plugins/crypto/server-crypto/server-crypto.helpers.ts`

GCM is an authenticated mode — in addition to encrypting, it produces an **auth tag** that guarantees integrity. That is why each field has its own `iv` + `tag`.

---

## Why are header and body encrypted separately?

The `GET /vault` (list) endpoint needs to return the **headers** of all user items, but **not the bodies** (which can be large and are not needed for listing).

This allows the server to decrypt (layer 2) only the header during a `list`, without touching the body — saving processing and avoiding unnecessary data exposure.

| Endpoint | Server decrypts | Returns to client |
|---|---|---|
| `GET /vault` | `encryptedHeader` only | header (L1 still encrypted) + `clientIv` |
| `GET /vault/:id` | `encryptedHeader` + `encryptedBody` | header + body (L1 still encrypted) + `clientIv` |

---

## Fields and their roles

| Field | DB type | Generated by | Purpose |
|---|---|---|---|
| `encrypted_header` | `bytea` | server (L2) | doubly encrypted header |
| `encrypted_body` | `bytea` | server (L2) | doubly encrypted body |
| `client_iv` | `bytea` | client | IV used by the client in L1 — required to decrypt on the client side |
| `server_header_iv` | `bytea` | server | L2 IV for the header — required for server decryption |
| `server_header_tag` | `bytea` | server | L2 auth tag for the header — ensures integrity (GCM) |
| `server_body_iv` | `bytea` | server | L2 IV for the body |
| `server_body_tag` | `bytea` | server | L2 auth tag for the body |

## Flow: Store (`POST /vault`)

```
Client                                     Server
   │                                          │
   │  encryptedHeader (base64, L1)            │
   │  encryptedBody   (base64, L1)   ──────►  │
   │  clientIv        (base64)                │
   │                                          │
   │                              L2 encrypt(encryptedHeader)
   │                                → serverHeaderIv + serverHeaderTag
   │                              L2 encrypt(encryptedBody)
   │                                → serverBodyIv + serverBodyTag
   │                                          │
   │                              INSERT into DB (bytea)
   │                                          │
   │  ◄──────────────────────────  { cuid }   │
```

**Code** (`vault.service.ts:14-28`):
```ts
const headerL2 = crypto.encrypt(Buffer.from(input.encryptedHeader, "base64"));
const bodyL2   = crypto.encrypt(Buffer.from(input.encryptedBody, "base64"));

vaultRepository.create({
  encryptedHeader: headerL2.encrypted,
  encryptedBody:   bodyL2.encrypted,
  clientIv:        Buffer.from(input.clientIv, "base64"),
  serverHeaderIv:  headerL2.iv,
  serverHeaderTag: headerL2.tag,
  serverBodyIv:    bodyL2.iv,
  serverBodyTag:   bodyL2.tag,
});
```

## Flow: Retrieve (`GET /vault/:id`)

```
Client                                     Server
   │                                          │
   │  GET /vault/:id        ──────────────►   │
   │                                          │
   │                              SELECT from DB
   │                              L2 decrypt(encryptedHeader, serverHeaderIv, serverHeaderTag)
   │                                → back to encryptedHeader (L1)
   │                              L2 decrypt(encryptedBody, serverBodyIv, serverBodyTag)
   │                                → back to encryptedBody (L1)
   │                                          │
   │  ◄──────  encryptedHeader (base64, L1)   │
   │           encryptedBody   (base64, L1)   │
   │           clientIv        (base64)       │
   │                                          │
   │  L1 decrypt(encryptedHeader, clientIv, userMasterKey)
   │  L1 decrypt(encryptedBody,   clientIv, userMasterKey)
   │    → original data
```

---

## Flow: List (`GET /vault`)

Same as retrieve, but **only the header** is decrypted (L2). The body is left untouched — it is not even read from the database for this endpoint.

```
Response per item:
{
  cuid,
  encryptedHeader,  ← L2 removed, L1 still present
  clientIv,         ← for the client to decrypt when needed
  createdAt
}
```

---

## Responsibility summary

| Responsible | Action |
|---|---|
| **Client** | Encrypts with the user's master key (L1), generates `clientIv` |
| **Server** | Re-encrypts with its own key (L2), stores everything as `bytea` |
| **Server (retrieve)** | Removes L2, returns data still in L1 to the client |
| **Client** | Removes L1 using `clientIv` + user's master key |
